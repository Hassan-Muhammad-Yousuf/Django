# PROJECT OVERVIEW
This is a Django monolith that mixes a learning sandbox (“ground”) with a real-ish e‑commerce API (“store”), user accounts (“core”), tagging/likes (generic relations), and operational add‑ons (Celery tasks, caching, logging, DRF, JWT auth). The main app exposes REST endpoints for products, collections, carts, orders, reviews, and product images, plus Django admin customizations and a small HTML landing page.

---
# PROJECT STRUCTURE
- `manage.py` — Entry point for Django management commands; defaults to `front.settings.dev`.
- `front/` — Project config and runtime entrypoints.
  - `front/settings/common.py` — Shared settings: apps, middleware, REST framework, auth/JWT, email, caching, logging.
  - `front/settings/dev.py` — Dev config: DEBUG, hardcoded SECRET_KEY, Postgres DB settings.
  - `front/settings/prod.py` — Prod config: DEBUG False, SECRET_KEY from env, empty ALLOWED_HOSTS.
  - `front/urls.py` — Root URL router; includes admin, core, ground, store, djoser, debug toolbar, silk.
  - `front/asgi.py` / `front/wsgi.py` — ASGI/WSGI entrypoints.
  - `front/celery.py` — Celery app configuration and autodiscovery.
- `core/` — Custom user model and minimal homepage.
  - `core/models.py` — Custom `User` extending `AbstractUser`, unique email.
  - `core/serializers.py` — Djoser overrides for user creation/current user.
  - `core/admin.py` — Custom User admin + overrides store Product admin to add tag editing.
  - `core/urls.py` — TemplateView for `core/templates/core/index.html`.
  - `core/templates/core/index.html` — Basic static homepage.
  - `core/static/core/style.css` — Styling for homepage.
  - `core/signals/handlers.py` — Receives order_created signal; prints order.
- `store/` — Main e‑commerce domain + API.
  - `store/models.py` — Promotions, collections, products, images, customers, orders, cart, reviews.
  - `store/serializers.py` — DRF serializers for all resources + custom cart/order flows.
  - `store/views.py` — DRF ViewSets for products, collections, carts, orders, etc.
  - `store/urls.py` — DRF routers + nested routers for reviews/images/cart items.
  - `store/admin.py` — Admin customization, filters, inline images, inventory actions.
  - `store/permissions.py` — Custom permission classes.
  - `store/filter.py` — django-filter FilterSet for Product.
  - `store/pagination.py` — Default page size for DRF.
  - `store/valiodators.py` — File size validator (typo in filename).
  - `store/signals/handlers.py` — Auto-create Customer on new User.
  - `store/migrations/` — DB schema history for store.
  - `store/tests/` — pytest + DRF tests for products/collections.
- `tags/` — Tagging system using generic relations.
  - `tags/models.py` — `Tag`, `TaggedItem`, custom manager.
  - `tags/admin.py` — Tag admin with search.
- `likes/` — Generic “likes” via ContentTypes.
  - `likes/models.py` — `LikedItems` generic relation.
- `ground/` — Sandbox app for experiments and demos.
  - `ground/views.py` — “HelloView” calling httpbin, logging, lots of commented ORM experiments.
  - `ground/task.py` — Celery task example.
  - `ground/templates/` — Simple HTML template and email template.
- `locustfiles/browse_products.py` — Locust performance test against store endpoints.
- `pytest.ini` — pytest config for Django settings.
- `static/`, `media/` — Static and uploaded media roots.
- `celerybeat-schedule.db` — Celery beat schedule database (checked in).

# ARCHITECTURE & DESIGN DECISIONS
- **Architecture pattern:** Django monolith with multiple apps (MVT: Model‑View‑Template) plus DRF for API endpoints.
- **API design:** DRF ViewSets + routers, including nested routers for product reviews/images and cart items (`store/urls.py`).
- **Auth:** JWT via `djangorestframework-simplejwt` and Djoser for auth endpoints (`front/settings/common.py`).
- **Async tasks:** Celery with Redis broker + beat schedule for periodic tasks (`front/celery.py`, `front/settings/common.py`, `ground/task.py`).
- **Caching:** Django cache backed by Redis (`front/settings/common.py`).
- **Admin emphasis:** Significant admin customizations for store operations (`store/admin.py`, `core/admin.py`).
- **Generic relations:** Tags and likes use `ContentType` + `GenericForeignKey` (`tags/models.py`, `likes/models.py`).
- **No dependency manifest:** No `requirements.txt`/`pyproject.toml` present; dependencies must be inferred from imports/settings.

**Dependencies (inferred) and why they exist**
- `Django` — core framework.
- `djangorestframework` — REST API serializers/viewsets/routers (`store/serializers.py`, `store/views.py`).
- `django-filter` — filter backends for product filtering (`store/filter.py`).
- `django-cors-headers` — CORS handling (`front/settings/common.py`).
- `djoser` — user registration/auth endpoints (`front/urls.py`, `core/serializers.py`).
- `djangorestframework-simplejwt` — JWT auth (`front/settings/common.py`).
- `django-debug-toolbar` — request debugging (`front/urls.py`, `front/settings/common.py`).
- `django-silk` — performance profiling (`front/urls.py`, `front/settings/common.py`).
- `whitenoise` — static file serving in production (`front/settings/common.py`).
- `celery` — background tasks (`front/celery.py`, `ground/task.py`).
- `redis` + `django-redis` — cache + broker (`front/settings/common.py`).
- `requests` — external HTTP calls in `ground/views.py`.
- `psycopg2` (or psycopg2‑binary) — Postgres DB backend (`front/settings/dev.py`).
- `pytest`, `pytest-django` — tests (`pytest.ini`, `store/tests/`).
- `model-bakery` — test data factories (`store/tests/`).
- `locust` — load testing (`locustfiles/browse_products.py`).
- Optional/commented: `templated-mail` (commented in `ground/views.py`).

# CORE CONCEPTS EXPLAINED
- **Django MVT (Model‑View‑Template)**
  - What: Django’s variation of MVC: Models handle data, Views handle logic, Templates handle UI.
  - Why: Standard Django structure.
  - How: Models in `store/models.py`, views in `store/views.py`, templates in `core/templates/...`.
  - Analogy: Kitchen (models), chef (views), plated dish (templates).
- **Django ORM**
  - What: Python API for DB queries.
  - Why: Avoid raw SQL, consistent model access.
  - How: `Product.objects...` in `store/views.py` and `ground/views.py`.
  - Analogy: Using a translator instead of speaking a foreign language directly.
- **DRF ViewSets + Routers**
  - What: Bundled CRUD endpoints with routing helpers.
  - Why: Rapid API creation with consistent URL patterns.
  - How: `ModelViewSet` in `store/views.py`, `DefaultRouter` in `store/urls.py`.
  - Analogy: A vending machine that exposes all buttons (list, create, update, delete) for a product.
- **Serializers**
  - What: Convert model instances to/from JSON and validate input.
  - Why: API input/output consistency and validation.
  - How: `ProductSerializer`, `CreateOrderSerializer` in `store/serializers.py`.
  - Analogy: A customs checkpoint translating and inspecting items in transit.
- **Nested Routers**
  - What: URLs for resources tied to parent resources.
  - Why: Scoped endpoints like `/products/{id}/reviews/`.
  - How: `rest_framework_nested` in `store/urls.py`.
  - Analogy: A subfolder inside a folder.
- **Permissions**
  - What: Rules controlling access to endpoints.
  - Why: Separate admin vs normal user actions.
  - How: `IsAdminOrReadOnly` in `store/permissions.py`, used in `store/views.py`.
  - Analogy: Staff‑only doors in a store.
- **JWT Authentication**
  - What: Token‑based auth for APIs.
  - Why: Stateless auth for API clients.
  - How: `SIMPLE_JWT` in `front/settings/common.py`, Djoser endpoints in `front/urls.py`.
  - Analogy: A wristband that grants access without re‑checking ID.
- **Signals**
  - What: Event hooks on model actions.
  - Why: Side effects like creating Customer or notifying on order.
  - How: `store/signals/handlers.py`, `core/signals/handlers.py`, `order_created` signal.
  - Analogy: A doorbell that triggers other devices.
- **Generic Relations**
  - What: One table linking to multiple model types using ContentTypes.
  - Why: Tags/likes for any model.
  - How: `tags/models.py`, `likes/models.py`.
  - Analogy: A universal adapter that fits many devices.
- **Celery Tasks + Beat**
  - What: Background job processing + scheduling.
  - Why: Async email or long tasks.
  - How: `front/celery.py`, `ground/task.py`, `CELERY_BEAT_SCHEDULE` in settings.
  - Analogy: A mailroom that handles deliveries while the front desk stays open.
- **Caching with Redis**
  - What: Store computed data in memory for speed.
  - Why: Reduce DB or external calls.
  - How: `CACHES` in `front/settings/common.py`, `cache` in `ground/views.py`.
  - Analogy: Keeping frequently used tools on the desk instead of the closet.
- **Django Admin Customization**
  - What: Tailoring admin UI with filters, inline edits, actions.
  - Why: Better data management for store staff.
  - How: `store/admin.py` and `core/admin.py`.
  - Analogy: A custom control panel instead of a generic one.

# FULL PROJECT WALKTHROUGH
1. **Entry point**
   - Management commands go through `manage.py` (settings `front.settings.dev`).
   - Web requests enter via `front/wsgi.py` (WSGI) or `front/asgi.py` (ASGI).
2. **Root routing**
   - `front/urls.py` routes:
     - `/admin/` → Django admin.
     - `/` → `core/urls.py` (static homepage).
     - `/ground/` → `ground/urls.py` (sandbox).
     - `/store/` → `store/urls.py` (main API).
     - `/auth/` → Djoser auth routes (JWT).
3. **Core homepage**
   - `core/urls.py` serves `core/templates/core/index.html`, styled by `core/static/core/style.css`.
4. **Ground sandbox**
   - `ground/urls.py` maps `/ground/hello/` to `HelloView`.
   - `ground/views.py` calls httpbin via `requests` and renders `ground/templates/index.html`.
5. **Store API**
   - **Products:** `ProductViewSet` in `store/views.py` handles list/create/update/delete, filtering, searching, pagination; uses `ProductSerializer`.
   - **Collections:** `CollectionViewSet` supports product counts and delete protection.
   - **Reviews:** `ReviewViewSet` uses nested router to scope reviews by product.
   - **Carts:** `CartViewSet` (create/retrieve/delete) and `CartItemViewSet` for line items.
   - **Customers:** `CustomerViewSet` with `/customers/me/` for current profile.
   - **Orders:** `OrderViewSet` supports creating orders from carts and admin‑only updates/deletes.
   - **Product images:** `ProductImageViewSet` handles uploads.
6. **Data models**
   - `store/models.py` defines the e‑commerce domain.
   - Relationships: Collection → Products, Orders → Items → Products, Cart → Items, Product → Images/Reviews.
7. **Signals**
   - New User creates Customer (`store/signals/handlers.py`).
   - Order creation emits `order_created`, consumed in `core/signals/handlers.py`.
8. **Auth**
   - Custom user model in `core/models.py`.
   - Djoser uses custom serializers in `core/serializers.py`.
9. **Background tasks**
   - `front/celery.py` configures Celery.
   - `ground/task.py` defines `notify_customer`.
10. **Testing**
   - pytest configured in `pytest.ini`, focused on products/collections (`store/tests/...`).
11. **Performance testing**
   - Locust script in `locustfiles/browse_products.py` simulates browsing and cart usage.

# KEY IMPLEMENTATION DETAILS
- **Hardcoded secrets and settings:** `SECRET_KEY` hardcoded in `front/settings/dev.py`; `ALLOWED_HOSTS` empty in `front/settings/prod.py`.
- **CORS origin uses https on localhost:** `CORS_ALLOWED_ORIGINS` in `front/settings/common.py` is `https://127.0.0.1:8001`; may need `http://` for local dev.
- **JWT auth header type:** `SIMPLE_JWT` uses `AUTH_HEADER_TYPES = ('JWT',)` not the default `Bearer`.
- **Media uploads:** `ProductImage.image` uses `validate_file_size` in `store/valiodators.py`; note typo in filename.
- **File size math:** `validate_file_size` uses `1004` not `1024` bytes per KB.
- **Order creation is transactional:** `CreateOrderSerializer.save()` uses `transaction.atomic()` to create order + items and delete cart.
- **Signals are partially wired:** `store/apps.py` loads `store.signals.handlers`, `core/apps.py` does not appear to load `core.signals.handlers` (currently imported in `core/apps.py` only if added; check if `core/apps.py` actually registers signals).
- **Potential missing permissions:** Many ViewSets rely on default permission classes (none set in settings), meaning public access unless a class explicitly sets permissions.
- **Error handling gaps:** `CustomerViewSet.me` and `OrderViewSet.get_queryset` assume a `Customer` exists; will raise if not created.
- **External call without timeout:** `requests.get` in `ground/views.py` lacks a timeout.
- **Git includes runtime artifacts:** `celerybeat-schedule.db` is checked in.
